# Спека: автоматизация индексации невыплаченного оклада

## Цель

Автоматически заполнить в Google Sheets колонку `Сумма индексации недоплаты на момент подачи иска (20.05.2026)` для каждой строки с месячной недоплатой.

Таблица:

`https://docs.google.com/spreadsheets/d/15GQcgRInpcISWHn6hXvH0b-iSziPpQngl_Ggh-3EQ7o/edit?usp=sharing`

Внешний калькулятор:

`https://calc.consultant.ru/ind`

Калькулятор КонсультантПлюс считает в браузере по JSON-справочникам:

- `https://calc.consultant.ru/conf/calc_ind/indexes.json`
- `https://calc.consultant.ru/conf/calc_ind/indexes_from_2022.json`
- `https://calc.consultant.ru/conf/calc_ind/regions.json`
- `https://calc.consultant.ru/conf/calc/calendar.json`
- `https://calc.consultant.ru/conf/calc_395/rates.json`

Скрипт должен обращаться к этим данным через `UrlFetchApp`. `calendar.json` используется как справочник нерабочих дней производственного календаря.
`rates.json` используется для расчета компенсации за задержку зарплаты по калькулятору `https://calc.consultant.ru/kompensaciya-zarplata`.

## Входные колонки

Скрипт по умолчанию обращается к фиксированным колонкам на листе:

### Лист `Оклад`

- `D` - год
- `E` - месяц
- `F` - `ИПЦ (к соответствующему месяцу предыдущего месяца)`
- `H` - `Проп. проинд. оклад`, используется как корректная сумма оклада для реконструкции годового заработка на листе `Отпуска`
- `J` - `Недоплата по окладу`
- `K` - `Сумма индексации недоплаты на момент подачи иска (20.05.2026)`
- `L` - `Пени за несвоевременную выплату (ст. 236 ТК РФ)`

Строка заголовков: `2`. Первая строка данных: `3`. Дата окончания индексации читается из `K2`. Дата окончания пеней читается из `L2`, если там есть дата; иначе используется дата из `K2`.

### Лист `Ежемесячные`

- `C` - `Месяц, год расчета`
- `G` - `Недоплата по ежемесячным премиям (с учетом фактически выплаченных )`
- `H` - `Сумма индексации недоплаты на момент подачи иска (20.05.2026)`
- `I` - `Пени за несвоевременную выплату (ст. 236 ТК РФ)`

Строка заголовков: `1`. Первая строка данных: `2`. Колонка `F` на этом листе не трогается, потому что в ней находятся фактически выплаченные премии. Дата окончания индексации читается из `H1`. Дата окончания пеней читается из `I1`, если там есть дата; иначе используется дата из `H1`.

### Лист `Ежеквартальные`

- `C` - `Квартал расчета и месяц выплаты`
- `G` - `Недоплата ежеквартальных премий (с учетом фактически выплаченных)`
- `H` - `Сумма индексации недоплаты на момент подачи иска (20.05.2026)`
- `I` - `Пени за несвоевременную выплату (ст. 236 ТК РФ)`

Строка заголовков: `1`. Первая строка данных: `2`. Дата окончания индексации читается из `H1`. Дата окончания пеней читается из `I1`, если там есть дата; иначе используется дата из `H1`.

### Лист `Ежегодные`

- `C` - `Год расчета и месяц выплаты`
- `G` - `Недоплата ежегодных премий (с учетом фактически выплаченных)`
- `H` - `Сумма индексации недоплаты на момент подачи иска (20.05.2026)`
- `I` - `Пени за несвоевременную выплату (ст. 236 ТК РФ)`

Строка заголовков: `1`. Первая строка данных: `2`. Дата окончания индексации читается из `H1`. Дата окончания пеней читается из `I1`, если там есть дата; иначе используется дата из `H1`.

### Лист `Отпуска`

- `A` - дата выплаты отпуска
- `B` - `Сумма корректного годового заработка`
- опционально отдельная колонка `Дата начала отпуска` - дата события для выбора расчетного 12-месячного периода
- `J` - `Недоплата по отпускным выплатам`
- `K` - `Сумма индексации недоплаты на момент подачи иска (20.05.2026)`
- `L` - `Пени за несвоевременную выплату (ст. 236 ТК РФ)`

Колонка `B` заполняется автоматически как сумма корректных начислений за 12 календарных месяцев, предшествующих месяцу даты начала отпуска. Если отдельной колонки `Дата начала отпуска` нет или она пуста, скрипт использует дату выплаты отпуска из `A` как fallback: `Оклад!H` + `Ежемесячные!B` + `Ежеквартальные!B` + `Ежегодные!B`, отфильтрованных по месяцу/периоду соответствующей строки.

Годовые премии при реконструкции отпускной базы берутся из `Ежегодные!B` только за завершенный календарный год. Если строка годовой премии относится к году, который на дату отпуска еще не прошел, она не включается в отпускной средний заработок. Пропорциональная годовая премия за неполный год остается расчетом для самостоятельного требования о взыскании премии, а не для отпускной базы.

### Служебные листы

Скрипт больше не создает лист `Методология`. Если такой лист остался от прежних запусков, следующий массовый пересчет удалит его вместе с устаревшим листом `Проверка`.

## Правило расчета

Для каждой строки:

1. Берется базовая сумма недоплаты: для листа `Оклад` из `Недоплата по окладу`, для листов премий и отпусков - из соответствующей колонки недоплаты.
2. Дата начала периода строится из `год` + `мес` на листе `Оклад` или из колонки расчетного периода на листах премий/отпусков.
3. Базовая дата начала: 5-е число месяца, следующего за месяцем из строки.
4. Если базовая дата начала выпадает на нерабочий день по производственному календарю, дата переносится влево на ближайшую рабочую дату.
5. Дата окончания: дата из скобок в заголовке целевой колонки, например `20.05.2026` в `Сумма индексации недоплаты на момент подачи иска (20.05.2026)`.
6. Если в заголовке дата без года, год берется из даты запуска скрипта в таймзоне таблицы. Если дата в заголовке не распознана, используется дата запуска.
7. Регион по умолчанию: `Российская Федерация`, id `1`, как первый регион в калькуляторе.
8. Для каждого месяца периода применяется ИПЦ из JSON КонсультантПлюс.
9. Первый/последний неполный месяц считается пропорционально количеству дней, как в калькуляторе.
10. Месяцы с ИПЦ меньше 100% для предметного расчета приводятся к 100%, чтобы индексация не уменьшала реальное содержание зарплаты.
11. Если по последним месяцам еще нет данных Росстата/КонсультантПлюс, скрипт считает только опубликованные месяцы и пишет примечание в ячейку. Такие месяцы не считаются ошибкой проверки: например, при расчете на `20.05.2026` ИПЦ за май 2026 не нужен и не включается в множитель, пока он не опубликован.

Фактически значение целевой ячейки:

`Базовая недоплата * (цепной множитель ИПЦ за период - 1)`

Месячные ИПЦ перемножаются цепным способом, как в калькуляторе ст. 208 ГПК РФ. Для неполного месяца используется месячный множитель:

`(100 + (ИПЦ - 100) * дни периода в месяце / дни в месяце) / 100`

## Результат

Скрипт записывает числовые значения в колонки `F`, `K` и `L` на листе `Оклад`, а на листах премий и отпусков - в соответствующие колонки индексации и пеней по настройкам листа. Для `F` используется формат коэффициента `0.0000`, для денежных колонок - денежный формат. Для пустых или итоговых строк формулы сохраняются.

## Помесячный ИПЦ

Для колонки `F` скрипт берет ИПЦ к предыдущему месяцу из данных КонсультантПлюс по `год` + `мес` и региону `Российская Федерация`.

В JSON значение хранится как процент, например `100.86`. В таблицу записывается коэффициент для формул:

`100.86 / 100 = 1.0086`

Если ИПЦ меньше 100%, например `99.60`, в таблицу записывается фактический коэффициент `0.9960`, а ячейка помечается фиолетовым. Только в расчете индексации такой коэффициент приводится к `1.0000`, чтобы не уменьшать сумму.

## Пени

Для колонки пеней применяется та же установленная дата выплаты, что и для индексации: 5-е число следующего месяца с переносом влево на ближайший рабочий день по производственному календарю.

Компенсация считается как в калькуляторе КонсультантПлюс:

`Общая недоплата по окладу * ключевая ставка / 100 / 150 * количество дней задержки`

На листах премий и отпусков база для пеней берется из колонки общей недоплаты соответствующего листа. Если база для пеней `<= 0`, пени записываются как `0`.

Период пеней начинается со дня, следующего за установленной датой выплаты, и заканчивается датой окончания расчета включительно. Если в периоде менялась ключевая ставка, расчет разбивается на интервалы ставок.

## Методологические примечания

### Дефляционный ИПЦ в цепной формуле

- В таблицу по колонке ИПЦ записывается фактический коэффициент из источника. Например, если ИПЦ равен `99,60%`, в ячейке остается `0,9960`.
- Для расчета индексации зарплаты дефляционный коэффициент не должен уменьшать сумму. Позиция: индексация по ст. 134 ТК РФ направлена на обеспечение повышения уровня реального содержания заработной платы в связи с ростом потребительских цен, поэтому месячный коэффициент для расчета принимается как `max(ИПЦ / 100, 1)`.
- Для расчета индексации самой недоплаты применяется та же позиция. Смысл требования - восстановить покупательную способность невыплаченной суммы, а не уменьшить уже возникшую задолженность из-за отдельного месяца с ИПЦ ниже `100%`.
- Практическая формула: `цепной множитель = произведение max(ИПЦ / 100, 1)`, `индексация = долг * (цепной множитель - 1)`.
- Если калькулятор КонсультантПлюс по ст. 208 ГПК РФ математически учитывает дефляционный месяц как коэффициент ниже `1`, результат калькулятора может быть ниже. Для трудового спора в этой автоматизации используется позиция: дефляция не увеличивает долг, но и не уменьшает его.

### Отпуска и средний заработок по ПП N 922 / ПП N 540

- Для отпусков используется логика среднего заработка по 12 календарным месяцам, предшествующим периоду сохранения среднего заработка. Если на листе есть колонка `Дата начала отпуска`, расчетный период реконструируется от нее; если нет - от даты выплаты отпуска в `Отпуска!A`.
- `Отпуска!B` сейчас заполняется как сумма корректных начислений из `Оклад!H`, `Ежемесячные!B`, `Ежеквартальные!B`, `Ежегодные!B` за 12 месяцев до месяца даты события. Это соответствует рабочему допущению, что премии учитываются по месяцу выплаты.
- Для годовых премий есть отдельный юридический нюанс: по п. 15 ПП N 922 и ПП N 540 вознаграждение по итогам работы за год, начисленное за предшествующий событию календарный год, учитывается независимо от времени начисления. В текущей автоматизации `Ежегодные!B` может содержать пропорциональную сумму для требования о взыскании премии за неполный год, но при расчете отпускных такая строка не включается, если календарный год премии еще не завершен на дату отпуска.
- Для квартальных и иных премий за период больше месяца текущий скрипт берет сумму `B` один раз по месяцу выплаты. Он не проверяет ограничения п. 15: не более одной выплаты за каждый показатель, пропорциональность при неполном расчетном периоде и специальные правила для премий за период, превышающий расчетный период.
- Скрипт заполняет числитель отпускного среднего заработка (`Отпуска!B`) и не валидирует делитель. Делитель по ПП N 922 / ПП N 540 для отпусков: `12 * 29,3`, а при неполных/исключаемых месяцах - корректированный делитель по правилам п. 10. Колонки с делителем и количеством дней должны быть проверены отдельно в самой таблице.
- ПП N 540 применяется к событиям с 01.09.2025; до этой даты действует ПП N 922. В коде переходная дата задана как `01.09.2025`.

### Выходное пособие

- Средний дневной заработок для выходного пособия считается иначе, чем для отпусков: сумма заработка за расчетный период делится на количество фактически отработанных рабочих дней. Делитель `29,3` для выходного пособия не применяется.
- По ПП N 540 выходное пособие рассчитывается как средний дневной заработок, умноженный на среднее количество рабочих дней, приходящихся на один месяц в году (`рабочие дни года / 12`). Для стандартного года это ориентировочно `247 / 12 = 20,58`, для високосного `248 / 12 = 20,67`.

## Среда выполнения

Расчет выполняется не локально, а в Google Apps Script, привязанном к таблице. После установки скрипта в `Расширения -> Apps Script` он запускается в облаке Google и доступен с любого устройства, где есть доступ к таблице.

## Импорт расчетных листков 1С:ЗУП

Добавлен отдельный импорт из папки Google Drive. Ссылка на папку берется из самой расчетной таблицы: скрипт сначала ищет ячейку с подписью `Исходные данные:` и Drive-ссылку рядом справа, затем fallback-ом ищет первую ссылку вида `https://drive.google.com/drive/folders/...` в документе. Если ссылка не найдена в таблице, используется `ZUP_IMPORT_SETTINGS.SOURCE_FOLDER_URL`.

`https://drive.google.com/drive/folders/1YpnqMHnY0K0ZwJIttm8aggzUGv3TBkpm`

Скрипт `ZupImport.gs` сканирует файлы в этой папке и подпапках, распознает Google Sheets, Google Docs, HTML, CSV и PDF. Если в папке есть несколько вариантов одного расчетного листка, импорт выбирает канонический источник в порядке: Google Docs, HTML, Google Sheets, CSV, PDF. Остальные варианты фиксируются в листе качества, но не дублируют строки.

PDF читается через временную OCR-конвертацию в Google Docs. Для этого в Apps Script должен быть включен Advanced Drive service (`Drive API`, версия `v2`) и scope `https://www.googleapis.com/auth/drive`; если сервис не включен, PDF не ломает импорт, а попадает в отчет качества с понятной причиной. XLS/XLSX по-прежнему нужно предварительно открыть/сохранить как Google Sheets или CSV.

Результат импорта пишется в служебные листы:

- `Импорт_1С_ЗУП` - нормализованные строки из расчетных листков: файл, лист, сотрудник, период, год, месяц, рабочие дни, оплачено дней, дата выплаты, ведомость, дата ведомости, раздел 1С, категория, вид начисления, начислено, выплачено, удержано, исходная строка.
- `Импорт_1С_Свод` - агрегированная сводка по файлу, месяцу, разделу 1С, категории и виду начисления: рабочие дни, оплачено дней, начислено, выплачено, удержано, количество строк.
- `Импорт_1С_Диагностика` - сверка импортированных начислений с текущими значениями на рабочих вкладках `Оклад`, `Ежемесячные`, `Ежеквартальные`, `Ежегодные`, `Отпуска`.
- `Импорт_1С_Качество` - отчет по каждому расчетному листку: выбранный файл, MIME, найденные дубли, количество строк, сотрудник, период, итоги разделов из источника, распознанные суммы и предупреждения.
- `Импорт_1С_Состояние` - технический индекс последнего импорта по группе файлов: id выбранного файла, дата изменения, размер, версия парсера, подпись и количество строк. Повторный запуск использует эти данные, чтобы не перечитывать неизменившиеся файлы без необходимости.

Для подготовки реконструкции из расчетных листков скрипт может создать рядом с рабочими вкладками отдельные вкладки структуры:

- `Из_1С_Оклад`
- `Из_1С_Ежемесячные`
- `Из_1С_Ежеквартальные`
- `Из_1С_Ежегодные`
- `Из_1С_Отпуска`

Эти вкладки копируют формат, заголовки и расчетные формулы соответствующих целевых вкладок, но очищают поля, которые должны быть восстановлены из расчетных листков или пересчитаны после реконструкции. Для `Оклад` очищаются `A`, `B`, `D`, `E`, `F`, `I`, `K`, `L`; для премиальных вкладок - период, фактически выплаченная сумма и расчетные итоги; для `Отпуска` - даты/дни отпуска, начисления из расчетных листков, отпускная база и расчетные итоги.

Категории распознаются эвристически по тексту строки:

- `Оклад`
- `Ежемесячные премии`
- `Ежеквартальные премии`
- `Ежегодные премии`
- `Отпуска`
- `Выходное пособие`
- `Удержания`
- `Доплата до оклада`
- `Командировки`
- `Больничные`
- `Отпуск без оплаты`
- `Материальная помощь`
- `Сверхурочные и ночные`
- `Праздники и выходные`
- `Районные коэффициенты и надбавки`
- `Совмещение и замещение`

Для выплат из расчетного листка отдельно выделяются номер/описание ведомости и дата ведомости. Например, строка `За первую половину месяца (Банк, вед. № 5 от 19.01.24)` дает вид выплаты `За первую половину месяца`, ведомость `Банк, вед. № 5 от 19.01.24` и дату `19.01.2024`.

Для строк начислений отдельно извлекаются рабочие дни и оплаченные дни, если они есть в расчетном листке рядом с периодом начисления.

Период строки импорта берется из периода самого расчетного листка/имени файла. Внутренние даты в строках начислений, отпусков и больничных остаются в описании строки, но не должны перекидывать весь расчетный листок в другой месяц.

Диагностика не перезаписывает рабочие вкладки. Она показывает, какие строки сходятся с импортом, какие не сходятся, и где не удалось найти импортированные данные для сопоставления. Для `Отпуска` диагностический импорт пытается восстановить 12-месячную базу из начислений перед датой события отпуска.

Лист качества сверяет распознанные суммы с итогами разделов `Начислено`, `Удержано`, `Выплачено`, если такие итоги удалось найти в структуре расчетного листка. Расхождения не блокируют импорт, но явно показываются в предупреждениях.

Для чтения папки Drive, Google Docs, внешних JSON-справочников и OCR PDF в `appsscript.json` нужны scopes `https://www.googleapis.com/auth/spreadsheets`, `https://www.googleapis.com/auth/documents`, `https://www.googleapis.com/auth/drive`, `https://www.googleapis.com/auth/drive.readonly`, `https://www.googleapis.com/auth/script.external_request`, а также зависимость Advanced Drive service.

## Запуск

В таблице появляется меню `Индексация недоплаты`:

- `Обновить активный лист` - пересчитать все строки активного листа.
- `Обновить все листы` - пересчитать строго последовательно: `Оклад`, `Ежемесячные`, `Ежеквартальные`, `Ежегодные`, `Отпуска`, затем удалить устаревшие служебные листы `Методология` и `Проверка`, если они есть.
- `Обновить ежемесячные премии` - пересчитать лист `Ежемесячные` напрямую, независимо от того, какой лист сейчас активен.
- `Обновить квартальные премии` - пересчитать лист `Ежеквартальные`.
- `Обновить ежегодные премии` - пересчитать лист `Ежегодные`.
- `Обновить отпуска` - пересчитать лист `Отпуска`.

Также появляется меню `Импорт 1С`:

- `Проверить импорт без перезаписи` - прочитать источники и обновить только отчет качества, не трогая `Импорт_1С_ЗУП`, `Импорт_1С_Свод`, `Импорт_1С_Диагностика` и состояние.
- `Импортировать расчетные листки` - прочитать папку Drive с выгрузками 1С и обновить `Импорт_1С_ЗУП` / `Импорт_1С_Свод`.
- `Полный импорт с перечитыванием` - принудительно перечитать все файлы, даже если их подпись не изменилась.
- `Создать вкладки структуры из 1С` - создать рядом с целевыми вкладками пустые структурные вкладки `Из_1С_*` для будущего заполнения из расчетных листков.
- `Очистить импорт 1С` - удалить листы импорта.
